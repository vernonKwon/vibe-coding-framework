# Playbook 작성 가이드

각 프로젝트 레포에 `PLAYBOOK.md`를 생성하면, Consultant Agent가 코드 스캔 시 이 파일을 참조하여 프로젝트에 맞는 기술 부채를 식별한다.

## 배치 위치

```
my-project/
├── CLAUDE.md      ← 코딩 컨벤션 (이미 있다면)
├── PLAYBOOK.md    ← 리팩터링 규칙 (이 가이드 참고하여 작성)
└── src/
```

## 작성 프롬프트

아래 프롬프트를 Claude에게 주면 프로젝트에 맞는 PLAYBOOK.md를 자동 생성할 수 있다:

```
이 프로젝트의 코드베이스를 분석해서 PLAYBOOK.md를 작성해줘.

PLAYBOOK.md는 자동화된 리팩터링 에이전트가 참조하는 파일이야.
에이전트는 이 파일을 읽고 코드에서 기술 부채를 식별한 후, GitHub Issue를 생성하고,
코드를 수정해서 Draft PR을 만든다.

다음 섹션을 포함해줘:

1. **프로젝트 컨텍스트** — 기술 스택, 아키텍처 패턴, 도메인 요약 (에이전트가 맥락을 이해하도록)
2. **식별 규칙** — 이 프로젝트에서 특별히 검사해야 할 패턴들
   - 우선순위별로 분류 (Critical / High / Medium / Low)
   - 가능하면 코드 패턴 예시 포함 (Bad → Good)
3. **금지 사항** — 에이전트가 절대 수정하면 안 되는 것
4. **테스트 방법** — 수정 후 검증에 사용할 명령어

주의:
- CLAUDE.md에 이미 정의된 코딩 컨벤션은 반복하지 마.
- 이 프로젝트만의 고유한 규칙에 집중해.
- 에이전트가 읽는 파일이므로 사람이 아닌 LLM이 이해하기 쉽게 작성해.
```

## PLAYBOOK.md 템플릿

```markdown
# [프로젝트명] Refactoring Playbook

## 프로젝트 컨텍스트

- 기술 스택: [예: Kotlin / Spring Boot 3.4 / JPA + QueryDSL / PostgreSQL]
- 아키텍처: [예: CQRS 패턴, Soft Delete 필수]
- 도메인: [예: 배터리 교환 관리 시스템]

## 식별 규칙

### Critical (즉시 처리)
- [예: Repository 쿼리에서 deletedAt.isNull 조건 누락]
- [예: 하드코딩된 시크릿]

### High (1주 내)
- [예: 50라인 초과 함수]
- [예: 테스트 없는 비즈니스 로직]

### Medium (2주 내)
- [예: any 타입 사용]
- [예: 사용되지 않는 코드]

### Low (백로그)
- [예: 주석이 필요한 복잡한 로직]

## 금지 사항

- DB 스키마/마이그레이션 변경 금지
- 외부 라이브러리 의존성 추가/제거 금지
- 기존 API 인터페이스 변경 금지
- .env, 시크릿 파일 수정 금지
- 이슈에 명시되지 않은 파일 수정 금지

## 테스트 방법

[빌드/테스트/린트 명령어 — .workspace.yaml의 commands와 동일]
```

## 공통 규칙

프로젝트 공통으로 적용할 규칙이 있다면 PLAYBOOK.md 상단에 포함하거나,
팀 위키에서 관리하고 PLAYBOOK.md에서 링크한다.

에이전트는 항상 다음 원칙을 따른다:
- **기능 보존**: 리팩터링은 동작을 변경하지 않는다
- **Atomic PR**: 하나의 PR은 하나의 논리적 변경만 포함
- **테스트 선행**: 테스트 없는 코드의 리팩터링은 금지
- **컨벤션 준수**: CLAUDE.md 규칙을 최우선으로 따른다
